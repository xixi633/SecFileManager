### 预览功能优化（第一阶段）

#### ✨ 新增
- 新增预览配置接口：`GET /file/preview/config`
  - 返回字段：`maxPreviewSize`、`smallFileThreshold`、`partialPreviewSize`、`maxRangeResponseSize`
- 前端新增获取预览配置 API：`fetchPreviewConfig()`
- 预览弹窗新增错误态展示：
  - 显示失败原因
  - 提供“重试 / 下载文件”操作

#### 🔧 优化
- 后端预览阈值改为统一配置化读取，不再依赖控制器中的硬编码常量：
  - 小文件阈值（完整加载）
  - 大文件阈值（拒绝预览）
  - 大文本部分预览大小（Range）
  - 单次 Range 最大响应大小
- 前端 `FilePreview` 改为启动时拉取后端预览配置，动态决定预览策略
- 大文件文本预览 Range 请求改为使用后端配置的 `partialPreviewSize`
- 预览关闭/切换时补充错误状态清理，避免旧错误残留

#### 🧩 配置项变更
在 `backend/src/main/resources/application.yml` 新增/明确以下配置：
- `secure-file.preview.small-file-threshold`
- `secure-file.preview.partial-preview-size`
- `secure-file.preview.max-range-response-size`
- 保留并继续使用：
  - `secure-file.preview.large-file-threshold`
  - `secure-file.preview.chunk-size`

#### 🗂 影响文件
- `backend/src/main/java/com/security/filemanager/controller/FileController.java`
- `backend/src/main/java/com/security/filemanager/service/FileService.java`
- `backend/src/main/resources/application.yml`

### 文件夹预览性能优化（第二阶段）

#### 🔧 优化
- 后端新增“文件夹解密结果缓存”（按 `userId + folderId` 维度），避免同一文件夹在短时间内重复全量解密
- 将文件夹相关高频路径接入缓存读取：
  - `listFolderEntries(...)`
  - `getFolderEntryDataWithLimit(...)`
  - `deleteFolderEntry(...)`
  - `browseFolderContent(...)`
- 删除/变更文件夹内容后，新增对应缓存失效逻辑，保证预览一致性

#### 🚀 效果
- 浏览文件夹后再预览子文件时响应更快
- 同一文件夹内连续预览多个文件时延迟显著下降
- 减少重复解密带来的 CPU 与 I/O 开销

#### 🗂 影响文件
- `backend/src/main/java/com/security/filemanager/service/FileService.java`

### 大文件上传链路重构（第三阶段）

#### 🔧 优化
- 后端分片上传改为“边接收边增量加密写入最终文件”，去除“上传后再全量读取分片合并”的二次流程
- 每个分片到达即完成：
  - 顺序校验
  - 原文哈希增量计算
  - 分片加密写入最终 `.enc` 文件
  - 追加分片认证标签
- 最后一个分片到达时直接完成落库（写入 `t_file` 元数据），无需长时间合并等待
- 前端大文件上传并发调整为顺序上传（`CONCURRENT_UPLOADS = 1`），与后端增量写入策略匹配
- 保留 `merge` 接口用于“完成提交”兼容，避免前端协议断裂

#### 🚀 效果
- “前面上传很快、最后合并卡很久”的现象显著缓解
- 上传总耗时分布更均匀，进度条更贴近真实处理进度
- 降低磁盘二次读取与临时分片合并开销

#### ⚠️ 注意
- 当前策略要求分片按顺序上传；若后续要恢复并发乱序上传，需要额外重排机制

#### 🗂 影响文件

### 大文件上传可靠性增强（第四阶段）

#### ✨ 新增
- 新增上传会话状态查询接口：`GET /file/chunk/session/status`
  - 返回 `stage / progress / nextChunkNumber / updatedAt` 等状态字段
- 新增上传会话重置接口：`POST /file/chunk/session/reset`
- 新增上传会话取消接口：`POST /file/chunk/session/cancel`
- 前端新增 API：
  - `fetchChunkSessionStatus(identifier)`
  - `resetChunkSession(identifier)`

#### 🔧 优化
- 后端分片会话增加状态机与可观测阶段：
  - `INIT / UPLOADING / FINALIZING / DONE / FAILED / EXPIRED`
- 增加会话超时与定时清理机制：
  - `session-timeout-ms`
  - `session-cleanup-interval-ms`
- 统一错误返回增加 `errorCode` 字段，前端按错误码映射更友好提示
- 前端接入断点续传恢复：
  - 上传前先查会话状态
  - 从 `nextChunkNumber` 继续上传
- 上传进度区新增“重传 / 取消”操作
- 取消上传时优化提示策略：不再弹“上传失败”，仅提示“已取消上传”

#### 🧩 配置项变更
在 `backend/src/main/resources/application.yml` 新增：

- `secure-file.upload.session-cleanup-interval-ms`

### 上传/文件夹打开性能与进度逻辑优化（第五阶段）

#### 🔧 优化
- 大文件上传并发从 `1` 提升到 `2`，提升带宽利用率
- 后端分片写入支持“乱序到达 + 顺序刷盘”：
  - 分片先进入会话内存缓冲
  - 按 `nextChunkNumber` 连续落盘
  - 兼容前端小并发上传
- `merge` 接口返回 `fileId`，前端可在上传成功后直接做增量更新
- 上传完成后列表采用“乐观插入 + 延迟校正刷新”，减少整页阻塞等待
- 文件夹浏览增加“同层目录预热”策略，提前缓存相邻目录 entries
- 文件夹浏览进度条改为“非假进度”模式：
  - 使用 `indeterminate` 流动条
  - 以真实阶段文案反馈状态（读取信息/请求目录/渲染目录）
  - 完成后展示实际耗时
- 文件夹上传压缩等级由 `DEFLATE level 6` 调整为 `level 1`，降低前端 CPU 压缩耗时

#### 🚀 效果
- 大文件上传平均耗时进一步下降，上传尾段等待感减少
- 上传成功后文件可更快出现在列表中，界面响应更及时
- 打开文件夹时不再展示伪百分比，进度反馈更真实
- 连续打开同层文件夹时，二次打开速度明显提升

#### 🗂 影响文件
- `backend/src/main/java/com/security/filemanager/service/FileChunkService.java`
- `backend/src/main/java/com/security/filemanager/controller/ChunkController.java`
- `frontend/src/api/file.js`
- `frontend/src/views/FileList.vue`
