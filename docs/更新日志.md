# 更新日志

## 2026-04-05

### kkFileView 联调稳定性与文件夹预览性能修复

#### ✨ 新增
- 新增文件夹安全预览接口：`GET /file/folder/preview-safe/{fileId}/{pathBase64}`，用于 `kkFileView` 回源场景，避免复杂路径参数在 query 中被截断。
- 后端新增 `secure-file.viewer.source-base-url` 配置，支持固定回源基地址，避免浏览器访问域名（如 `localhost`）污染 `kkFileView` 回源链接。
- 预览弹窗新增“预览方式”切换：`自动 / 原生预览 / kkFileView`，支持用户随时切换。

#### 🔧 优化
- 前端 API 基础地址由硬编码 `localhost` 改为运行时动态计算：
  - 优先读取 `VITE_API_BASE_URL`
  - 未配置时按当前访问主机自动拼接 `:8080/api`
- `>2GB` 文件策略调整为“仅允许 kkFileView 预览”，并同步放开后端预览接口对 `>2GB` 的硬拒绝逻辑，改为流式输出。
- 自动模式下，文件夹内条目默认优先原生预览，减少 `kkFileView` 在容器回源链路下的首开卡顿。

#### 🚀 性能与可用性修复
- 文件夹内大条目预览新增“内存缓存 + 磁盘缓存”双层策略：
  - 小条目使用内存缓存
  - 大条目落盘缓存并支持 Range 随机读取
  - 避免每次拖动进度条都重复解压同一条目
- 鉴权拦截器放行 `preview-safe` 路径的 `token` 参数认证，修复 `kkFileView` 回源 `401` 问题。
- 文件夹条目路径包含中文、空格、`&`、引号等特殊字符时，`kkFileView` 预览不再因 query 参数歧义而失败。

#### 🧩 配置项变更
在 `backend/src/main/resources/application.yml` 新增/明确：
- `secure-file.viewer.source-base-url`

#### 🗂 影响文件
- `backend/src/main/java/com/security/filemanager/controller/FileController.java`
- `backend/src/main/java/com/security/filemanager/interceptor/AuthInterceptor.java`

## 2026-04-04

### 核心办公文档与高级预览引擎升级 (kkFileView 集成)

#### ✨ 新增
- 完整集成开源高性能预览引擎 `kkFileView` 作为系统的默认“高级查看器”。
- **全量接管策略**：突破了过去仅有部分白名单文件（如 Word/PDF 等）能被内置或第三方支持预览的局限，现在平台针对任意被点击的文件都会默认调用高级查看器服务。
- **内置预览服务源码**：将其可用版本的源码直接合入工程团队仓库中，以后无需安装和配置 Docker，拉取项目用 IDEA 就能一键把 `kkFileView` 跑起来联调。

#### 🔧 优化
- **URL Base64安全解析**：改进后端预览回调地址的生成策略，采用安全规范的 `{base64Url}` 模板配合 `&fullfilename` 文件名后缀投递，彻底解决 `kkFileView v4+` 版本的违规拦截告警和无法精准识别文件格式的痛点。
- **自动错误降级 (Auto-Fallback)**：在前端 `FilePreview.vue` 引擎中添加 iframe 无感探测机制，当拦截到诸如“暂不支持此格式/正在生成页面失败/404页面”等高级预览错误信号时，由系统自动平滑回退（降级）为自带的原生渲染服务（如图片查看、代码 Monaco 编辑器等）。
- **完善本地部署说明**：同步更新文档，增加了不用装 Docker 的 IDEA 原生一键启动步骤指引。

#### 🛡️ 容错体验
- 在所有高级查看器页面常驻了**「切换为原生基础预览」**兜底提示栏。当处于异构网络环境导致无法自动探测内部报错时，用户仍可顺滑点击实现手动降级。

#### 🗂 影响文件
- `backend/src/main/resources/application.yml`
- `backend/src/main/java/com/security/filemanager/controller/FileController.java`
- `frontend/src/components/FilePreview.vue`
- `docs/本地部署指南.md`

### 文件夹预览性能优化（第二阶段）

- 后端新增“文件夹解密结果缓存”（按 `userId + folderId` 维度），避免同一文件夹在短时间内重复全量解密

### 大文件上传链路重构（第三阶段）

#### 🔧 优化
- 后端分片上传改为“边接收边增量加密写入最终文件”，去除“上传后再全量读取分片合并”的二次流程
- 每个分片到达即完成：
  - 顺序校验
  - 原文哈希增量计算
  - 分片加密写入最终 `.enc` 文件
  - 追加分片认证标签
- 最后一个分片到达时直接完成落库（写入 `t_file` 元数据），无需长时间合并等待
- 前端大文件上传并发调整为顺序上传（`CONCURRENT_UPLOADS = 1`），与后端增量写入策略匹配
- 保留 `merge` 接口用于“完成提交”兼容，避免前端协议断裂

#### 🚀 效果
- “前面上传很快、最后合并卡很久”的现象显著缓解
- 上传总耗时分布更均匀，进度条更贴近真实处理进度
- 降低磁盘二次读取与临时分片合并开销

#### ⚠️ 注意
- 当前策略要求分片按顺序上传；若后续要恢复并发乱序上传，需要额外重排机制

### 上传/文件夹打开性能与进度逻辑优化（第五阶段）

#### 🔧 优化
- 大文件上传并发从 `1` 提升到 `2`，提升带宽利用率
- 后端分片写入支持“乱序到达 + 顺序刷盘”：
  - 分片先进入会话内存缓冲
  - 按 `nextChunkNumber` 连续落盘
  - 兼容前端小并发上传
- `merge` 接口返回 `fileId`，前端可在上传成功后直接做增量更新
- 上传完成后列表采用“乐观插入 + 延迟校正刷新”，减少整页阻塞等待
- 文件夹浏览增加“同层目录预热”策略，提前缓存相邻目录 entries
- 文件夹浏览进度条改为“非假进度”模式：
  - 使用 `indeterminate` 流动条
  - 以真实阶段文案反馈状态（读取信息/请求目录/渲染目录）
  - 完成后展示实际耗时
- 文件夹上传压缩等级由 `DEFLATE level 6` 调整为 `level 1`，降低前端 CPU 压缩耗时

#### 🚀 效果
- 大文件上传平均耗时进一步下降，上传尾段等待感减少
- 上传成功后文件可更快出现在列表中，界面响应更及时
- 打开文件夹时不再展示伪百分比，进度反馈更真实
- 连续打开同层文件夹时，二次打开速度明显提升

#### 🗂 影响文件
- `backend/src/main/java/com/security/filemanager/service/FileChunkService.java`
- `backend/src/main/java/com/security/filemanager/controller/ChunkController.java`
