## 二、安全机制设计

### 2.1 文件加密机制

#### 2.1.1 加密算法选择

**算法**：AES-256-GCM

**选择理由**：
1. **安全性**：
   - AES-256 是 NIST 批准的标准算法
   - 256 位密钥长度，抵抗量子计算攻击
   - GCM 模式提供 AEAD（认证加密）

2. **性能**：
   - 硬件加速支持（AES-NI 指令集）
   - 并行计算，适合大文件加密
   - 无需 padding，减少计算开销

3. **安全特性**：
   - 自动完整性校验
   - 防止 padding oracle 攻击
   - TLS 1.3 默认使用

#### 2.1.2 加密流程

```
输入：原始文件 + 文件密钥 + IV
      ↓
【AES-256-GCM 加密】
      ↓
输出：密文 + 认证标签（Auth Tag）
```

**关键参数**：
- **密钥长度**：256 位（32 字节）
- **IV 长度**：96 位（12 字节，GCM 推荐长度）
- **Auth Tag**：128 位（16 字节）

**安全保证**：
- ✅ 机密性：密文无法直接还原
- ✅ 完整性：Auth Tag 保证数据未被篡改
- ✅ 真实性：Auth Tag 验证数据来源

#### 2.1.3 IV（初始化向量）管理

**生成方式**：
```java
SecureRandom secureRandom = new SecureRandom();
byte[] iv = new byte[12]; // GCM推荐12字节
secureRandom.nextBytes(iv);
```

**存储方式**：
- 存储在数据库（Hex 编码）
- 每个文件独立 IV
- 与密文分开存储

**为什么需要 IV**：
- 相同明文 + 相同密钥 → 不同密文
- 防止频率分析攻击
- GCM 模式下，IV 不可重复使用

### 2.2 密钥管理机制

#### 2.2.1 三层密钥架构

本系统采用严格的三层密钥体系（Encryption Envelope），确保数据安全性。

```
┌───────────────────────────────────────┐
│  Tier 1: 系统主密钥 (System Master Key) │
│  - 存储：系统配置文件 / 环境变量          │
│  - 用途：加密用户主密钥                  │
│  - 算法：AES-256                        │
└─────────────────┬─────────────────────┘
                  │ 
                  ↓ 加密 (WRAP)
┌───────────────────────────────────────┐
│  Tier 2: 用户主密钥 (User Master Key)   │
│  - 存储：数据库 (以密文形式存储)          │
│  - 用途：加密该用户的文件密钥            │
│  - 特性：每用户独立，注册时随机生成        │
└─────────────────┬─────────────────────┘
                  │ 
                  ↓ 加密 (WRAP)
┌───────────────────────────────────────┐
│  Tier 3: 文件密钥 (File Encryption Key) │
│  - 存储：数据库 (以密文形式存储)          │
│  - 用途：加密文件内容                    │
│  - 特性：每文件独立，上传时随机生成        │
└─────────────────┬─────────────────────┘
                  │ 
                  ↓ 加密 (ENCRYPT)
            【文件内容密文】
```

#### 2.2.2 密钥生命周期管理

1.  **生成**：使用 `SecureRandom` 强随机数生成器产生 256 位密钥。
2.  **存储**：
    *   **EK (Encryption Key)** 永远不以明文形式持久化存储。
    *   **KEK (Key Encryption Key)** 仅在内存中短暂停留，使用完毕立即清除。
3.  **使用**：解密操作仅在内存中进行，严禁写入磁盘交换区。
4.  **销毁**：用户注销时，销毁其用户主密钥，导致下层所有文件密钥不可解密，实现由于密码学保证的数据销毁（Crypto-shredding）。

#### 2.2.3 密钥层次的优势

1. **密钥轮换**：
   - 更换用户主密钥：只需重新加密文件密钥记录（Metadata re-encryption）。
   - 无需重新加密GB级的文件内容。
   - 降低密钥轮换的计算成本和IO开销。

2. **权限隔离**：
   - 用户间密钥完全独立。
   - 即使数据库泄露（其中包含用户主密钥密文和文件密钥密文），攻击者若无系统主密钥，无法解开任何数据。
   - 实现了纵深防御（Defense in Depth）。

## 三、系统流程与安全说明（重点）

### 3.1 注册与登录流程

1. **注册**：
     - 用户提交用户名/密码
     - 后端生成用户主密钥（256 位）并使用系统主密钥加密存储
     - 密码以安全散列方式存储（禁止明文）

2. **登录**：
     - 用户凭证校验成功后签发 JWT
     - 前端持有 JWT，用于后续 API 访问
     - 服务端所有文件相关接口必须鉴权并绑定用户身份

**安全要点**：
- JWT 保护会话，避免明文会话标识
- 用户主密钥只在服务端解密使用

### 3.2 上传与加密保存流程

```
原始文件
    ↓
计算 SHA-256（完整性校验）
    ↓
可选压缩（节省空间）
    ↓
生成文件密钥（DEK）
    ↓
AES-256-GCM 加密
    ↓
密文写入本地存储
    ↓
文件密钥用用户主密钥加密后入库
```

**大文件分块加密**：
- 超过上传阈值时启用分块加密
- 每块独立 Auth Tag，防止块级篡改
- 分块头包含版本/块大小/IV 基础信息

**安全要点**：
- 每文件独立密钥 + 独立 IV
- AES-GCM 提供机密性与完整性
- 文件 Hash 用于完整性校验与追踪

### 3.3 下载与解密流程

1. 校验用户权限与文件归属
2. 读取密文文件与加密文件密钥
3. 解密文件密钥（用户主密钥）
4. AES-256-GCM 解密得到原始内容
5. 触发下载并记录下载信息

**安全要点**：
- 任何越权访问都会被拦截
- GCM Auth Tag 校验防篡改

### 3.4 预览流程（不影响下载安全）

- 预览本质是受控的解密读取
- 超过预览阈值的文件不允许在线预览
- 所有预览接口同样受 JWT 鉴权

### 3.5 回收站与删除流程

- 删除默认逻辑删除（可恢复）
- 彻底删除时同时清理密文文件
- 保留审计信息（下载次数、时间等）

### 3.6 安全性总结

**核心保障**：
- **AES-256-GCM**：机密性 + 完整性
- **三层密钥体系**：系统主密钥 → 用户主密钥 → 文件密钥
- **文件级随机密钥 + 随机 IV**：避免密文关联
- **SHA-256 完整性校验**：检测传输/存储篡改
- **严格鉴权与用户隔离**：防止越权访问

**结果**：
- 即使数据库与存储泄露，攻击者仍无法还原文件内容
- 攻击需同时获取系统主密钥、用户主密钥与文件密钥，成本极高

3. **密钥托管**：
   - 支持密钥恢复机制（扩展功能）
   - 支持多用户共享文件（扩展功能）

### 2.3 完整性校验机制

#### 2.3.1 双重校验架构

```
┌─────────────────────────────┐
│    GCM Auth Tag 校验         │
│    - 校验对象：加密数据       │
│    - 校验时机：解密时         │
│    - 失败处理：拒绝解密       │
└──────────────┬──────────────┘
               ↓
┌─────────────────────────────┐
│    SHA-256 哈希校验          │
│    - 校验对象：原始文件       │
│    - 校验时机：解密后         │
│    - 失败处理：拒绝返回       │
└─────────────────────────────┘
```

#### 2.3.2 第一层：GCM Auth Tag

**工作原理**：
```
加密时：
  Cipher.doFinal(plaintext)
  → ciphertext + auth_tag

解密时：
  Cipher.doFinal(ciphertext + auth_tag)
  → 自动验证 auth_tag
  → 若不匹配，抛出 AEADBadTagException
```

**防御能力**：
- ✅ 检测密文篡改
- ✅ 检测密钥错误
- ✅ 检测 IV 篡改
- ✅ 防止比特翻转攻击

**代码实现**：
```java
try {
    byte[] plaintext = cipher.doFinal(ciphertext);
    // 解密成功，Auth Tag 校验通过
} catch (AEADBadTagException e) {
    // 数据被篡改或密钥错误
    throw new RuntimeException("文件完整性校验失败");
}
```

#### 2.3.3 第二层：SHA-256 哈希

**存储时机**：
```java
// 上传时计算原始文件哈希
byte[] originalFile = file.getBytes();
String fileHash = AESUtil.sha256(originalFile);
// 存入数据库
```

**校验时机**：
```java
// 下载后重新计算哈希
byte[] decryptedFile = decrypt(...);
String computedHash = AESUtil.sha256(decryptedFile);

if (!computedHash.equals(storedHash)) {
    throw new RuntimeException("文件哈希不匹配");
}
```

**防御能力**：
- ✅ 验证解密正确性
- ✅ 检测应用层错误
- ✅ 提供审计记录

#### 2.3.4 为什么需要双重校验？

| 校验层 | 校验对象 | 检测场景 |
|-------|---------|---------|
| GCM Auth Tag | 加密数据 | 传输/存储中的篡改 |
| SHA-256 | 原始文件 | 解密过程错误 |

**场景示例**：
- 场景1：攻击者修改加密文件 → GCM 校验失败
- 场景2：密钥错误导致解密出错误内容 → SHA-256 校验失败
- 场景3：解密代码 Bug → SHA-256 校验失败

### 2.4 密码安全机制

#### 2.4.1 密码存储方案

**算法**：PBKDF2-HMAC-SHA256

**参数配置**：
```java
algorithm: PBKDF2WithHmacSHA256
iterations: 210,000  // OWASP 2023推荐
salt_length: 16 bytes (128 bits)
output_length: 32 bytes (256 bits)
```

#### 2.4.2 注册流程

```
用户输入密码
    ↓
生成随机盐值（16字节）
    ↓
PBKDF2(password, salt, 210000)
    ↓
密码哈希（32字节）
    ↓
存入数据库：
  - password_hash
  - password_salt
```

**代码实现**：
```java
// 生成盐值
String salt = PasswordUtil.generateSalt();

// 计算哈希
String hash = PasswordUtil.hashPassword(password, salt);

// 存入数据库
user.setPasswordSalt(salt);
user.setPasswordHash(hash);
```

#### 2.4.3 登录验证流程

```
用户输入密码
    ↓
查询数据库获取：
  - stored_hash
  - stored_salt
    ↓
PBKDF2(input_password, stored_salt, 210000)
    ↓
computed_hash
    ↓
常量时间比较：
  computed_hash == stored_hash
    ↓
验证成功/失败
```

#### 2.4.4 防止时序攻击 (Timing Attack Prevention)

**原理说明**：

时序攻击是一种侧信道攻击（Side-channel attack）。当系统在比较两个敏感字符串（如哈希值）时，如果使用普通的字符串比较（如 Java 的 `String.equals`），系统会在发现第一个不同字符时立即返回 `false`。

- 如果前 5 位相同，系统可能在 10ns 后返回。
- 如果前 6 位相同，系统可能在 12ns 后返回。

攻击者通过精确测量响应时间，一位一位地猜测，最终可以还原出完整的哈希值。

**解决方案：常量时间比较 (Constant Time Comparison)**

本系统实现的 `PasswordUtil.constantTimeEquals` 采用“累计异或”的方式：
1. 无论输入内容如何，始终完整遍历两个字符串的所有字节。
2. 使用 `result |= a[i] ^ b[i]` 累积差异。
3. 循环结束后，再一次性判断 `result` 是否为 0。

这构建了一道“时间墙”，确保比较操作的耗时只与数据长度有关，而与数据内容无关，从而彻底防御时序攻击。

**代码实现**：

```java
// ✅ 安全：构建“时间墙”，耗时固定
private static boolean constantTimeEquals(String a, String b) {
    // ... 长度检查 ...
    
    int result = 0;
    // 必须跑完所有循环，绝不提前 return
    for (int i = 0; i < aBytes.length; i++) {
        result |= aBytes[i] ^ bBytes[i];
    }
    return result == 0;
}
```

#### 2.4.5 盐值的作用

**没有盐值的问题**：
```
用户A：password123 → hash_value
用户B：password123 → hash_value (相同)

攻击者可以：
1. 识别相同密码
2. 使用彩虹表批量破解
```

**使用盐值**：
```
用户A：password123 + salt_A → hash_A
用户B：password123 + salt_B → hash_B (不同)

攻击者必须：
1. 针对每个用户单独破解
2. 无法使用预计算的彩虹表
```

### 2.5 权限控制机制

#### 2.5.1 身份认证

**JWT Token 方案**：
```
登录成功 → 生成 JWT Token
    ↓
Token 结构：
{
  "sub": "1",           // 用户ID
  "username": "zhangsan",
  "iat": 1706000000,    // 签发时间
  "exp": 1706086400     // 过期时间
}
    ↓
HMAC-SHA256签名
    ↓
返回给客户端
```

**请求验证**：
```
客户端请求
    ↓
提取 Header: Authorization
    ↓
验证 Token 签名
    ↓
提取 user_id
    ↓
存入 ThreadLocal
    ↓
业务逻辑使用
```

#### 2.5.2 用户隔离

**数据库层隔离**：
```sql
-- ✅ 正确：强制带 user_id 条件
SELECT * FROM t_file 
WHERE id = #{fileId} 
  AND user_id = #{userId};

-- ❌ 错误：缺少 user_id 条件
SELECT * FROM t_file 
WHERE id = #{fileId};
```

**应用层隔离**：
```java
// Controller 提取当前用户ID
Long userId = AuthInterceptor.getCurrentUserId();

// Service 强制传入 userId
FileInfo file = fileMapper.selectByIdAndUserId(fileId, userId);

if (file == null) {
    // 文件不存在 或 无权限
    throw new RuntimeException("无访问权限");
}
```

#### 2.5.3 密钥隔离

**架构保证**：
```
用户A的文件密钥 ← 用户A的主密钥加密
用户B的文件密钥 ← 用户B的主密钥加密

用户A无法解密用户B的文件密钥
→ 即使获取加密文件，也无法解密内容
```

**攻击场景分析**：
```
假设攻击者获取了：
1. 用户B的加密文件（从磁盘）
2. 用户B的加密文件密钥（从数据库）

但攻击者没有：
- 用户B的主密钥

结果：
→ 无法解密文件密钥
→ 无法解密文件内容
→ 攻击失败
```

## 三、安全性证明

### 3.1 机密性分析

**定理**：未授权用户无法获取文件明文内容。

**证明**：
1. 文件以 AES-256-GCM 加密存储
2. 文件密钥使用用户主密钥加密
3. 用户主密钥使用系统主密钥加密
4. 系统主密钥不在数据库中

**攻击路径分析**：
- 路径1：直接读取加密文件 → 需要文件密钥 → 需要用户主密钥 → 需要系统主密钥 → ❌
- 路径2：SQL注入获取密钥 → 参数化查询防护 → ❌
- 路径3：越权访问API → 强制user_id校验 → ❌

**结论**：机密性得到保证。

### 3.2 完整性分析

**定理**：文件篡改必然被检测。

**证明**：
1. GCM Auth Tag 在解密时自动校验
2. SHA-256 哈希在返回前二次校验
3. 任一校验失败即拒绝访问

**攻击路径分析**：
- 修改加密文件 → GCM校验失败 → ❌
- 修改Auth Tag → GCM校验失败 → ❌
- 修改密钥 → 解密失败 → ❌
- 修改哈希值 → SHA-256校验失败 → ❌

**结论**：完整性得到保证。

### 3.3 可用性分析

**威胁**：拒绝服务攻击、数据丢失。

**防护措施**：
- 文件大小限制（application.yml配置）
- 数据库事务保证一致性
- 异常处理防止系统崩溃

**局限性**（课程设计不实现）：
- 未实现限流
- 未实现负载均衡
- 未实现备份恢复

### 3.4 认证强度分析

**密码破解成本计算**：

假设：
- 密码空间：$62^8 \approx 2.18 \times 10^{14}$
- PBKDF2 单次验证成本约 $100\text{ms}$（高迭代）

则穷举时间约为：

$$
2.18 \times 10^{14} \times 0.1\text{s}
= 2.18 \times 10^{13}\text{s}
\approx 6.92 \times 10^5 \text{年}
$$

结论：在合理密码策略下，离线暴力破解成本极高。

## 四、聊天模块安全设计（一期）

### 4.1 安全目标

1. 仅好友可建立会话和收发消息。
2. 消息内容入库加密，防止明文泄露。
3. 聊天文件下载必须通过授权校验，不能绕过文件所有者隔离。
4. WebSocket 握手必须携带 JWT。

### 4.2 鉴权与授权

- REST 接口复用 `AuthInterceptor` 获取当前用户。
- WebSocket 通过 `ChatHandshakeInterceptor` 校验 JWT。
- 会话访问前检查“当前用户是否属于该会话”。
- 消息发送前检查“双方是否为好友”。

### 4.3 消息存储安全

- 文本消息使用 AES-GCM 加密后入库，存储字段：
    - `content_ciphertext`
    - `content_iv`
    - `content_auth_tag`
- 发送者身份以服务端登录态为准，忽略客户端伪造 senderId。
- 使用 `(session_id, client_msg_id)` 唯一键保证幂等，避免重试导致重复消息。

### 4.4 聊天文件传输安全

- 复用已有文件加密存储体系，不新增独立文件明文链路。
- 发送文件消息时写入 `t_chat_file_share` 授权记录，包含：
    - 所有者 `owner_user_id`
    - 接收方 `receiver_user_id`
    - 过期时间 `expires_at`
    - 撤销标记 `revoked`
- 下载聊天文件时，先校验授权，再调用原有文件解密下载流程。

### 4.5 一期边界

- 已实现：好友一对一、实时推送、离线补拉、文件授权下载。
- 未实现：端到端加密、群聊密钥协商、多节点消息总线。

假设：
- 密码：8位，包含大小写字母+数字
- 可能性：62^8 ≈ 2.18 × 10^14

使用 PBKDF2，每次哈希耗时约 **100ms**（210,000次迭代）

**暴力破解时间**：
```
总时间 = 2.18 × 10^14 × 100ms
       = 2.18 × 10^13 秒
       ≈ 692,000 年
```

**使用彩虹表**：
- 由于每个用户独立盐值
- 需要针对每个用户单独计算
- 彩虹表优势失效

**结论**：认证强度足够。

## 七、安全验证指南 (Verification Guide)

本章节提供了一套标准化的验证流程，用于向用户或评估人员证明系统的安全性设计已真实落地，而非仅停留在文档层面。

### 7.1 验证密码存储安全性 (Password Privacy)

**目标**：证明数据库中不存储明文密码，且相同密码存储为不同哈希。

**步骤**：
1.  **准备**：注册两个用户 (UserA, UserB)，均设置密码为 `TestPassword123`。
2.  **检查**：查询数据库 `t_user` 表。
3.  **预期结果**：
    *   `password_hash` 字段看起来是随机乱码。
    *   UserA 和 UserB 的 `password_salt` **不同**。
    *   尽管明文密码相同，UserA 和 UserB 的 `password_hash` **完全不同**。
    *   **结论**：即使数据库泄露，攻击者无法直接获取密码，也无法通过彩虹表批量反查。

### 7.2 验证文件加密存储 (Encryption at Rest)

**目标**：证明磁盘上的文件是加密的，直接读取无法获取内容。

**步骤**：
1.  **准备**：上传一个包含敏感内容（如 "Secret Project 2024"）的文本文件 `secret.txt`。
2.  **定位**：在 `storage/` 目录下找到最新生成的物理文件（文件名可能是 UUID）。
3.  **检查**：使用文本编辑器（Notepad/VS Code）直接打开该物理文件。
4.  **预期结果**：
    *   文件内容显示为乱码（Encrypt Data）。
    *   全文搜索 "Secret" 关键词，**无法找到**。
    *   **结论**：服务器磁盘被物理窃取，数据依然安全。

### 7.3 验证完整性校验 (Integrity Check)

**目标**：证明系统能检测到即便是 1 个字节的数据篡改。

**步骤**：
1.  **准备**：上传一个文件，确保能正常下载预览。
2.  **破坏**：使用 Hex Editor (二进制编辑器) 打开 `storage/` 下对应的加密文件，随意修改其中 **1个字节** 的 16 进制值，保存。
3.  **检查**：尝试在前端下载或预览该文件。
4.  **预期结果**：
    *   请求失败，前端提示“文件验证失败”或“下载出错”。
    *   后端日志中出现 `AEADBadTagException` (GCM Tag 校验失败) 或 `RuntimeVerifyException` (Hash 校验失败)。
    *   **结论**：任何对加密数据的恶意修改都会被 GCM 算法识别并拒绝解密。
